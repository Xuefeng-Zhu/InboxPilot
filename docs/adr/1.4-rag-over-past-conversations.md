# 1.4 — RAG over Past Resolved Conversations: Implementation Plan

## 1. Goals & non-goals

**Goals**
- When the AI processes a new inbound message, surface **similar past resolved/escalated conversations** alongside KB chunks in the LLM context
- The AI can quote / model its answer on "here's how we handled this last time"
- Archive is built automatically on resolve and on escalate; backfilled for existing data; cleaned up on GDPR delete
- Multi-tenant safe (RLS, organization-scoped)

**Non-goals (v1)**
- Cross-encoder re-ranking (use a cheap LLM re-rank call)
- User-facing "similar conversations" UI in the inbox (this is an *AI context* feature, surfaced indirectly via the AI draft)
- Per-tenant opt-in toggle for archive or for using the archive in context (always on; per-tenant disable is a v1.1 follow-up)

## 2. Locked decisions

| # | Decision | Value |
|---|---|---|
| 1 | Summary text | **LLM-generated** structured JSON `{ issue, resolution, sentiment, topics }`. Rendered to a small narrative template for embedding. |
| 2 | PII handling | **Redact** emails / phones / cards / SSN in the source text before embedding. Original `messages` unchanged. |
| 3 | Recency weights | **1.0 / 0.8 / 0.5 / 0.0** at 30d / 180d / 2y cutoffs, hard floor 2y. |
| 4 | Archive scope | **Both `resolved` and `escalated`** trigger an archive. Re-trigger on every transition into either. Upsert overwrites. |
| 5 | Retrieval combination | **LLM re-rank**: pool of 14 (top-7 KB + top-7 archive) → gpt-4o-mini re-rank call → top-5 fed to the main LLM. No cache. |
| 6 | AI mode behavior | **Always include** archive context when matches exist (mode is the only switch). |
| 7 | Summary failure | **Fall back** to the deterministic template. Archive still happens. |
| 8 | Re-rank failure | **Fall back** to simple concat (top-5 KB + top-5 archive, ordered by similarity). |

## 3. Design overview

```
Inbound message
   │
   ▼
AiAgentService.processMessage (existing flow)
   │
   ├── generate embedding for latest message        (existing)
   │
   ├── match knowledge chunks          ──▶ top-7 KB candidates
   │
   ├── match past conversations        ──▶ top-7 archive candidates  ◀── NEW
   │      (uses match_conversation_archive RPC, same embedding)
   │
   ├── re-rank via gpt-4o-mini         ──▶ top-5 indices            ◀── NEW
   │      (fail-soft: falls back to simple concat top-5 from each)
   │
   ├── escalation engine (unchanged)
   │
   └── buildPrompt:
         ├─ knowledge chunks   (top-5)
         └─ past conversations (top-5)
            → system-prompt suffix: "When past conversations are relevant,
               reference them as examples. Do not invent details."
```

```
On resolve OR escalate (multiple sites)
   │
   ├── status = 'resolved' or 'escalated'    (existing)
   │
   └── enqueue 'archive_conversation_for_rag'   ◀── NEW
         │
         ▼
      process-jobs claims it
         │
         ▼
      ConversationArchiveService.archive(conversationId, orgId)
         1. Load conversation + messages (full history, capped)
         2. Generate LLM summary:
              aiClient.chatCompletion({ structured JSON output })
              on failure → buildDeterministicSummary(...)
         3. RedactPII(summaryText)
         4. Generate embedding
         5. Upsert into conversation_archive_embeddings
         6. audit_log: 'conversation_archived_for_rag'
```

## 4. Data model (migration `insforge/migrations/007_conversation_archive.sql`)

### 4.1 New table `conversation_archive_embeddings`

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK DEFAULT gen_random_uuid() | |
| `organization_id` | uuid NOT NULL FK → organizations | tenant scope for RLS |
| `conversation_id` | uuid NOT NULL UNIQUE FK → conversations | one row per conversation; upsert on re-archive |
| `summary_text` | text NOT NULL | the redacted text that was embedded |
| `summary_structured` | jsonb NULL | raw LLM output `{ issue, resolution, sentiment, topics }`; NULL when fallback template was used |
| `summary_source` | text NOT NULL CHECK `('llm','deterministic_fallback')` | which path produced the summary |
| `embedding` | vector(1536) NOT NULL | same dim as `knowledge_chunks` |
| `channel` | text NOT NULL CHECK (`('sms','email','webchat')`) | filterable |
| `tags` | text[] NOT NULL DEFAULT '{}` | from latest `ai_decisions.tags` on this conversation (best-effort) |
| `resolution_kind` | text NOT NULL CHECK `('agent_resolved','ai_resolved','auto_closed','escalated')` | filterable; `escalated` = no resolution yet |
| `resolved_by` | text NULL | user id of the agent who resolved/escalated, if known |
| `message_count` | integer NOT NULL | sanity / debug |
| `resolved_at` | timestamptz NOT NULL | when the conversation entered the terminal state (used for recency) |
| `created_at` | timestamptz NOT NULL DEFAULT now() | |
| `updated_at` | timestamptz NOT NULL DEFAULT now() | |

**Indexes**
- `idx_conversation_archive_embedding` HNSW on `(embedding vector_cosine_ops)` — same as KB
- `idx_conversation_archive_org_resolved` on `(organization_id, resolved_at DESC)` — for backfill queries
- UNIQUE on `(conversation_id)` — already implied; explicit

**RLS** — same pattern. SELECT/INSERT/UPDATE/DELETE gated by `organization_id IN (SELECT user_org_ids())`.

**Cascade** — when the parent `conversations` row is hard-deleted (GDPR), the archive row is removed via `ON DELETE CASCADE`.

### 4.2 New RPC `match_conversation_archive`

Modeled on `match_knowledge_chunks` (002):

```sql
CREATE OR REPLACE FUNCTION match_conversation_archive(
  query_embedding vector(1536),
  match_org_id uuid,
  match_limit int DEFAULT 7,
  match_threshold float DEFAULT 0.65,
  recency_floor timestamptz DEFAULT (now() - interval '2 years')
)
RETURNS TABLE (
  id uuid,
  conversation_id uuid,
  summary_text text,
  channel text,
  tags text[],
  resolution_kind text,
  resolved_at timestamptz,
  similarity float,
  recency_weight float
)
LANGUAGE plpgsql AS $fn$
BEGIN
  RETURN QUERY
  SELECT
    cae.id,
    cae.conversation_id,
    cae.summary_text,
    cae.channel,
    cae.tags,
    cae.resolution_kind,
    cae.resolved_at,
    (1 - (cae.embedding <=> query_embedding))::float * cae.recency_weight AS similarity,
    cae.recency_weight
  FROM (
    SELECT cae2.*,
      CASE
        WHEN cae2.resolved_at > now() - interval '30 days'  THEN 1.0
        WHEN cae2.resolved_at > now() - interval '180 days' THEN 0.8
        WHEN cae2.resolved_at > now() - interval '2 years'  THEN 0.5
        ELSE 0.0
      END AS recency_weight
    FROM conversation_archive_embeddings cae2
    WHERE cae2.organization_id = match_org_id
      AND cae2.resolved_at > recency_floor
  ) cae
  WHERE (1 - (cae.embedding <=> query_embedding)) * cae.recency_weight > match_threshold
  ORDER BY cae.embedding <=> query_embedding
  LIMIT match_limit;
END;
$fn$;
```

`recency_weight` is multiplied into the similarity projection AND surfaced in the result so the prompt builder can show it to the LLM. Hard floor at 2y — anything older never surfaces.

### 5.1 `packages/support-core/src/utils/pii-redactor.ts`

```ts
export function redactPii(text: string): string {
  return text
    .replace(EMAIL_RE, '[EMAIL]')
    .replace(PHONE_RE, '[PHONE]')
    .replace(CARD_RE, '[CC]')
    .replace(SSN_RE, '[SSN]')
    .trim();
}
```

Regexes are conservative (high-precision). False positives are acceptable; false negatives are not.

### 5.3 `packages/support-core/src/services/context-reranker.ts`

```ts
export class ContextRerankerService {
  constructor(private aiClient: AiClient) {}

  async reRank(params: {
    latestMessage: string;
    knowledgeCandidates: KnowledgeChunk[];
    archiveCandidates: MatchConversationArchiveResult[];
    model?: string;          // default 'openai/gpt-4o-mini'
    poolSize?: number;       // default 7 from each source
    outputSize?: number;     // default 5
  }): Promise<{ indices: number[]; model: string }> {
    // 1. Take top `poolSize` from each source (already pre-sorted by similarity).
    // 2. Build the candidate list: each item is { i, source, text, metadata }.
    // 3. System prompt:
    //    "You are a relevance ranker. Given a customer message and a numbered
    //     list of candidate context items, return a JSON array of the N most
    //     relevant item numbers, ordered most-relevant first. Consider: how
    //     directly each item addresses the customer's question, and how recent
    //     it is. Return ONLY the JSON array, no commentary."
    // 4. User prompt: customer message + the 14 candidate blocks.
    // 5. Parse the response (must be JSON array of integers).
    // 6. Validate indices are in range, de-dup, truncate to outputSize.
    // 7. Return { indices, model }.
    // Throws on parse/validation failure; caller falls back to top-N-by-similarity.
  }
}
```

### 6.2 `ConversationArchiveService` (new)

```ts
class ConversationArchiveService {
  constructor(
    private archiveRepo: ConversationArchiveRepository,
    private conversationRepo: ConversationRepository,
    private messageRepo: MessageRepository,
    private aiClient: AiClient,
    private auditLog: AuditLogRepository,
    private summaryModel: string,    // injected default; can be overridden per call
  ) {}

  async archiveConversation(conversationId, orgId): Promise<ConversationArchiveEmbedding> {
    // 1. Load conversation; assert status ∈ {'resolved','escalated'}; bail if not.
    // 2. Load messages (all, then cap to last 20 for the LLM call).
    // 3. Pick first message where senderType === 'contact' and last message
    //    where senderType ∈ {'user','ai','system'}.
    // 4. Determine resolutionKind:
    //    - 'agent_resolved' if a 'user' message exists in the last 3 messages
    //    - 'ai_resolved' if the resolution message was 'ai' (auto-reply, then resolve)
    //    - 'escalated' if status === 'escalated'
    //    - 'auto_closed' otherwise
    // 5. resolvedBy: latest 'user' sender_id if any, else null.
    // 6. Tags: from the latest ai_decisions row, if any.
    // 7. Try generateLlmSummary(...) — catch any error and fall back to buildDeterministicSummary(...).
    // 8. Generate embedding via aiClient.createEmbedding().
    // 9. archiveRepo.upsert(...) with summarySource: 'llm' or 'deterministic_fallback'.
    // 10. audit_log: 'conversation_archived_for_rag' with metadata.summarySource.
  }
}
```

### 6.3 `AiAgentService` (extend)

Insert a parallel retrieval step + a re-rank step after the KB match:

Extend `buildPrompt()` to include the re-ranked context:

```ts
private buildPrompt(
  messages,
  knowledgeChunks,
  archiveMatches,
  systemPrompt,
): ChatMessage[] {
  const system = [
    systemPrompt,
    '\n\n## Knowledge base context',
    ...(knowledgeChunks.length === 0
      ? ['(no KB matches above threshold)']
      : knowledgeChunks.map(c => `- ${c.content}`)),
    '\n\n## Past similar conversations (resolved or escalated)',
    'When the customer issue is similar to a past conversation, you may reference its resolution as an example. Do not invent details — only use what is provided.',
    ...(archiveMatches.length === 0
      ? ['(no similar past conversations found)']
      : archiveMatches.map(m =>
          `- [conv #${m.conversationId.slice(0,8)}] (${m.channel}, ${m.resolutionKind}, resolved ${m.resolvedAt.toISOString().slice(0,10)}, similarity ${m.similarity.toFixed(2)}):\n${m.summaryText}`
        )
    ),
  ].join('\n');

  return [
    { role: 'system', content: system },
    ...messages.map(m => ({ role: m.senderType === 'contact' ? 'user' : 'assistant', content: m.body })),
  ];
}
```

### 7.4 Triggers (multiple sites)

The archive job is enqueued on every transition to `resolved` or `escalated`. The trigger sites are:

1. **`app/api/functions/resolve-conversation/route.ts`** — after the existing `status='resolved'` update, enqueue the job. Already exists; just add the enqueue.

2. **`app/api/functions/escalate-conversation/route.ts`** — after `status='escalated'`, enqueue the job.

3. **`packages/support-core/src/services/ai-agent-service.ts` pre-LLM escalation path** — when the `EscalationEngine` triggers (`needs_human` branch, around line 168-200 of the current code), the service already updates the conversation. Add a `jobQueue.enqueue('archive_conversation_for_rag', ...)` call there.

4. **`app/api/functions/approve-ai-draft/route.ts`** — when approving an AI draft whose `decision_type === 'escalate'`, after the existing update, enqueue the job.

5. **Reopen path** — `app/api/functions/reopen-conversation/route.ts` is unchanged. No enqueue. The next resolve/escalate will trigger a re-archive; the upsert handles it.

In all sites, the enqueue is fire-and-forget relative to the HTTP response. If it fails, the user's action still succeeds; the archive simply lags and will be picked up by a follow-up re-archive on the next state transition.

## 8. Backfill

`scripts/backfill-conversation-archive.ts` (new, runnable via `tsx`):

1. Query `SELECT id, organization_id FROM conversations WHERE status IN ('resolved','escalated') AND id NOT IN (SELECT conversation_id FROM conversation_archive_embeddings)` in batches of 100.
2. For each row, insert a `support_jobs` row of type `archive_conversation_for_rag` with the right `organization_id`.
3. Let the existing `process-jobs` claim and process them.
4. Log progress: `Backfilled X / Y conversations (status=resolved|escalated)`.
5. Idempotent — safe to re-run.

Add an npm script:

```json
"scripts": {
  "backfill:archive": "tsx scripts/backfill-conversation-archive.ts"
}
```

## 9. GDPR / hard delete

When a conversation is hard-deleted, the `ON DELETE CASCADE` on `conversation_archive_embeddings.conversation_id` removes the row. The archive service is defensive: it loads the conversation and bails (silently) if not found, so a race between delete and archive-enqueue is harmless.

## 14. Acceptance criteria

- Resolving OR escalating a conversation enqueues an archive job; the job runs within ~30s; a row appears in `conversation_archive_embeddings` with a redacted `summary_text` and a 1536-dim vector.
- The `summary_structured` JSON has the four required fields when the LLM path succeeds; `summary_source='llm'` is set in that case.
- When the LLM summary call fails, the archive row is still created with `summary_source='deterministic_fallback'` and the deterministic template body.
- Re-resolving or re-escalating a conversation updates the existing row in place (no duplicate `conversation_id` rows).
- When the AI processes a new inbound message, the prompt includes a "Past similar conversations" section whose items came from a re-rank of the candidate pool.
- The re-rank call uses `gpt-4o-mini`-class model and returns top-5 indices; on re-rank failure, the prompt falls back to top-5 KB + top-5 archive by similarity.
- A conversation older than 2 years is never surfaced (recency floor in the RPC).
- The embedded `summary_text` contains no raw emails, phone numbers, credit-card patterns, or SSNs (PII redactor works).
- GDPR hard-delete of a conversation removes its archive row (cascade).
- `npm test` passes; `npm run lint` passes; the backfill script completes against the seed data without errors; the HNSW index on `conversation_archive_embeddings` is in place.
- Cost sanity: for a 1,000-inbound/day org, the additional LLM calls (summary + re-rank) stay under ~$0.50/day at gpt-4o-mini pricing.
