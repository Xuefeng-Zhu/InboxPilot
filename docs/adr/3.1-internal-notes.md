# 3.1 — Internal Notes & @mentions: Implementation Plan

## 1. Goals & non-goals

**Goals**
- Agents leave **private notes** on a conversation that the customer never sees
- **@mention** teammates to ping them; mentioned agent gets a real-time notification + a persistent bell-icon count
- Notes surface in the existing inbox thread, in their own right-sidebar tab
- All multi-tenant safe (RLS, RBAC, audit logged)

**Non-goals (v1)**
- Threaded replies to notes (flat list)
- Persistent notification table (realtime + bell count only)
- Cross-org mentions
- Notes on anything other than conversations (no notes on KB docs, contacts, etc.)
- Markdown rendering beyond `@` mentions and basic line breaks

## 2. Locked decisions

| # | Decision | Effect on plan |
|---|---|---|
| 1 | **Realtime-only mentions** | No `notifications` table, no `mentions` API route. Bell counter is an in-memory client state reset on bell-open. Migration 006 has only one new table. |
| 2 | **Flat list, no threading** | `parent_note_id` column added (unused) for future-proofing. No thread UI in v1. |
| 3 | **Soft delete with `deleted_at`** | Renders as `[note deleted]` placeholder. Preserves audit trail. |
| 4 | **@mention display = InsForge user email** | `parseMentions` matches `@<email>` tokens. Mention picker shows emails. No new column on `organization_members`. |
| 5 | **Replace "Audit" tab with "Notes"** | Right sidebar becomes `Insight \| Customer \| Notes`. The current "Audit" stub goes away; audit logs move to a dedicated page later (idea 6.2). |

## 3. Design overview

```
Agent A opens conversation
  │
  │ types in the composer (mode = "Note")
  │   "@jane@acme.com can you take a look at this refund?"
  │
  │ submits → POST /api/functions/internal-notes
  │   body: { conversationId, body: "@jane@acme.com can you…" }
  │
  ▼
Next.js route (internal-notes/route.ts)
  │ 1. JWT verify
  │ 2. RBAC: require reply_conversations
  │ 3. InternalNoteService.createNote(...)
  │    a. Parse body for @mentions → resolve @email → user_id via organization_members
  │    b. Insert internal_notes row
  │    c. audit_logs entry: 'internal_note_created'
  │ 4. Realtime publish on org:{orgId}:
  │    - event 'note_created'  → { note }
  │    - for each mentioned user: event 'mention' → { noteId, conversationId, actorId, recipientId }
  │ 5. Return note
  │
  ▼
Agent B (mentioned)
  │ realtime 'mention' event → toast "Jane mentioned you in #123"
  │ realtime 'note_created' event → if Agent B is on the conversation, render the note
  │ bell count increments
```

Notes are **never** written to the `messages` table, **never** sent to a provider, and **never** included in any customer-facing payload. The customer only ever sees `sender_type ∈ {contact, ai, user}` messages; the `MessageBubble` component is untouched.

## 4. Data model (migration `insforge/migrations/006_internal_notes.sql`)

### 4.1 New table `internal_notes`

| Column | Type | Notes |
|---|---|---|
| `id` | uuid PK DEFAULT gen_random_uuid() | |
| `organization_id` | uuid NOT NULL FK → organizations | tenant scope for RLS |
| `conversation_id` | uuid NOT NULL FK → conversations | which conversation this note belongs to |
| `author_id` | text NOT NULL | InsForge auth user id (matches `organization_members.user_id`) |
| `body` | text NOT NULL | raw text including `@email` tokens; rendered as markdown-light |
| `mentions` | text[] NOT NULL DEFAULT '{}' | resolved `user_id`s parsed from the body at write time |
| `parent_note_id` | uuid NULL FK → internal_notes | **reserved, not used in v1** (flat list) |
| `edited_at` | timestamptz NULL | set if body changed after creation |
| `deleted_at` | timestamptz NULL | soft-delete tombstone |
| `created_at` | timestamptz NOT NULL DEFAULT now() | |
| `updated_at` | timestamptz NOT NULL DEFAULT now() | |

**Indexes**
- `idx_internal_notes_conversation_created` on `(conversation_id, created_at DESC)` — primary read path: "load all notes for a conversation, newest first"
- `idx_internal_notes_org_mentions` GIN on `(organization_id, mentions)` — for "mentions me" filter
- `idx_internal_notes_author` on `(organization_id, author_id, created_at DESC)` — for "my notes" view if we add it

**RLS** — append to the same migration. SELECT/INSERT/UPDATE/DELETE on `internal_notes` gated by `organization_id IN (SELECT user_org_ids())`.

### 4.2 No `notifications` table

Realtime-only bell counter. No persistence required for v1.

### 4.3 support-core type changes

- `SenderType`: **no change** (notes are not messages)
- New entity type: `InternalNote`
- New input types: `CreateInternalNoteInput`, `UpdateInternalNoteInput`
- New util: `parseMentions(body, orgMembers) → { mentionedUserIds, cleanedBody }`

## 5. New services in support-core

### 5.1 `InternalNoteService` (new, `packages/support-core/src/services/internal-note-service.ts`)

```ts
class InternalNoteService {
  constructor(
    private noteRepo: InternalNoteRepository,
    private conversationRepo: ConversationRepository,
    private memberRepo: MemberRepository,
    private auditLog: AuditLogRepository,
    private realtime: RealtimePublisher,
  ) {}

  async createNote(orgId, authorId, conversationId, body): Promise<InternalNote> {
    // 1. Verify conversation exists and belongs to orgId
    // 2. Load org members (id + email)
    // 3. parseMentions(body, orgMembers) → resolved mentionedUserIds + cleanedBody
    // 4. Insert note (noteRepo.create)
    // 5. audit_log: 'internal_note_created'
    // 6. realtime.publish(`org:${orgId}`, 'note_created', { note, conversationId })
    // 7. For each mentioned user, publish 'mention' on same channel:
    //    { noteId, conversationId, actorId: authorId, recipientId }
    // 8. Return note
  }

  async listNotes(orgId, conversationId): Promise<InternalNote[]>
  async updateNote(orgId, authorId, noteId, body): Promise<InternalNote> {
    // 1. Load note, assert org match
    // 2. RBAC: author only
    // 3. Re-parse mentions on the new body
    // 4. Update (set edited_at)
    // 5. audit_log: 'internal_note_updated'
    // 6. realtime.publish note_updated
  }
  async deleteNote(orgId, authorId, noteId): Promise<void> {
    // 1. Load note
    // 2. RBAC: author OR (admin/owner role)
    // 3. Soft delete: set deleted_at = now()
    // 4. audit_log: 'internal_note_deleted'
    // 5. realtime.publish note_deleted
  }
}
```

### 5.2 `parseMentions` util (`packages/support-core/src/utils/mention-parser.ts`)

Pure function. Input: body string + list of `{ userId, email }`. Output: `{ mentionedUserIds: string[], cleanedBody: string }`.

Behavior:
- Match `@<email>` tokens. Email is matched literally (case-insensitive) against the member list.
- A token matches when `@<email>` in the body equals `@<member.email>` for any member (case-insensitive).
- Returns unique `user_id`s in the order they appear.
- The `cleanedBody` is the same as the input — we **do not** rewrite tokens; the renderer re-resolves for display.

> We could use a richer syntax (e.g. `@[Name](userId)`) but it adds friction. Plain `@email` matching is the right v1.

### 5.3 New repository

`packages/support-core/src/repositories/internal-note-repository.ts`:
- `findById(id)`
- `listByConversation(conversationId)` — filters out `deleted_at IS NOT NULL`
- `listByOrgAndMentions(orgId, userId)` — for "Mentions me" filter (`mentions cs {userId}`), filters out `deleted_at IS NOT NULL`
- `create(input)`
- `update(id, { body, mentions, editedAt })`
- `softDelete(id, deletedAt)`

## 6. New API routes (Next.js, all JWT-authenticated)

Mirror the existing `/api/functions/*` pattern (`getUserFromToken` from `_auth.ts`).

### 6.1 `app/api/functions/internal-notes/route.ts` (POST)
- Body: `{ conversationId, body }`
- 401 if no JWT, 403 if `!hasPermission(role, 'reply_conversations')`
- Delegates to `InternalNoteService.createNote`
- Returns `{ status: 'ok', data: { note } }`

### 6.2 `app/api/functions/internal-notes/[id]/route.ts`
- **GET** (list for a conversation, via query `?conversationId=...`) — returns notes (excluding soft-deleted)
- **PATCH** (update) — body: `{ body }` — author only
- **DELETE** (soft delete) — author or admin/owner
- Each action writes an `audit_logs` entry

## 7. Realtime events

Extend the documentation in `packages/support-core/src/interfaces/realtime-publisher.ts` (no code change to the publisher itself):

```ts
/**
 * Channel naming: `org:{organizationId}`
 *
 * Events:
 * - new_message — message inserted
 * - conversation_updated — status/ai_state changed
 * - knowledge_document_updated
 * - note_created    (NEW) { note, conversationId }
 * - note_updated    (NEW) { note }
 * - note_deleted    (NEW) { noteId, conversationId, deletedBy }
 * - mention         (NEW) { noteId, conversationId, actorId, recipientId }
 */
```

`useRealtime` in `lib/use-realtime.ts` is extended with new optional callbacks (`onNoteCreated`, `onNoteUpdated`, `onNoteDeleted`, `onMention`). The existing `useRealtime` callers in `MessageThread.tsx` opt-in.

The `mention` event is org-scoped (not per-user). The client filters: only increment the bell counter if `payload.recipientId === currentUser.id`. This avoids a per-user channel and keeps the auth model simple.

## 8. UI

### 8.1 ReplyComposer mode toggle — `components/inbox/ReplyComposer.tsx`

- Add a tab/toggle above the textarea: `[ Reply | Note ]`
- "Note" mode:
  - Placeholder: "Leave a private note for your team…"
  - Submit button label: "Post note"
  - Same `@` mention picker (shared component)
  - Different submit target: `POST /api/functions/internal-notes`
- "Reply" mode: identical to today
- Default mode: Reply. Mode remembered in component state per-conversation.

### 8.2 `@MentionPicker` — new component `components/inbox/MentionPicker.tsx`

- Floating popover anchored to the caret position (or a simple bottom-anchored dropdown in v1 to keep it simple)
- Triggered when the user types `@` in either composer mode
- Source: `useTeamMembers()` joined with the current user's email from `insforge.auth.getCurrentUser()` (the team page only has `user_id`; we need email from auth for display)
- Renders each member as `<email> · <role>`
- On select, inserts `@<email>` at the caret and closes
- Closes on Esc, click outside, or backspace

### 8.3 InternalNotesPanel — new component `components/inbox/InternalNotesPanel.tsx`

Renders in the right sidebar's **Notes** tab. (The right sidebar currently has `insight | customer | audit` — we replace "audit" with "notes".)

UI:
- Reverse-chronological list of notes
- Each note card: author avatar (initials from email), author email or display label, timestamp, body (with `@email` mentions highlighted as chips), edited indicator, kebab menu (Edit / Delete) when current user is author or admin
- Soft-deleted notes render as a faded `[note deleted]` placeholder
- Empty state: "No internal notes yet — leave one for your team."
- "Add note" button at the top (sets the composer to Note mode and focuses it)
- Live-updates via `useRealtime` — invalidate query on `note_created` / `note_updated` / `note_deleted`

### 8.4 MentionBell — new component `components/layout/MentionBell.tsx`

- Sits in the top of the sidebar (above nav)
- Bell icon + unread count badge
- Click → dropdown of latest mentions: each row is a tiny note preview + conversation id, click → jump to that conversation
- Realtime `mention` events bump the counter (client-side filter: only count if `payload.recipientId === currentUser.id`)
- Counter resets to 0 when the user opens the dropdown
- Hidden for `viewer` role (no point showing a bell they can't act on)

### 8.5 "Mentions me" inbox filter — `components/inbox/InboxFilters.tsx`

- New filter chip: "Mentions me"
- Sets `filters.mentionsMe = currentUserId`
- `useConversations` is extended: when set, do a two-step query:
  1. `SELECT DISTINCT conversation_id FROM internal_notes WHERE $userId = ANY(mentions) AND deleted_at IS NULL`
  2. `SELECT * FROM conversations WHERE id IN (...)` with the existing filters
- The two-step keeps the PostgREST query simple; the result set is naturally small (only conversations with notes mentioning the user).

## 9. RBAC

Permissions are unchanged. Behavior:
- **Read** notes: requires `view_conversations`
- **Create** notes: requires `reply_conversations`
- **Edit** notes: author only
- **Delete** notes: author OR a member with `manage_org` (owner/admin)

Server enforces these on every API route. No new permission enum values.

## 13. Acceptance criteria

- An agent with `reply_conversations` can post a note from the composer; the customer never sees it (no `messages` row created, no provider call, no realtime event on a customer-facing channel).
- `@jane@acme.com` in a note body resolves to that user's `user_id` and produces a `mention` realtime event targeted at them.
- The mentioned user sees a toast + their bell count increments within ~1s.
- A viewer-role user gets 403 on `POST /api/functions/internal-notes`; the bell is hidden for them.
- A note's author can edit/delete; an admin can delete any note; a viewer can only read.
- Deleting a note shows `[note deleted]` placeholder, the row stays in the table, and the audit log shows the deletion event with actor + timestamp.
- "Mentions me" inbox filter returns only conversations with a non-deleted note mentioning the current user.
- `npm test` passes; `npm run lint` passes; `internal_notes.mentions` is queryable via PostgREST `.contains('mentions', \`{${userId}}\`)` for the bell fallback.
