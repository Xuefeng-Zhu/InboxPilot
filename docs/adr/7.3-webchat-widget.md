# 7.3 ŌĆö Web Chat Widget: Implementation Plan

## 1. Goals & non-goals

**Goals**
- Org-scoped, drop-in JS snippet customers embed on their site (`<script src="ŌĆ”/widget.js" data-widget-id="ŌĆ”"></script>`)
- Live chat between an anonymous ŌåÆ identified visitor and an InboxPilot agent
- Reuse the existing AI pipeline, RAG, escalation, job queue, and audit logs without forking them
- Multi-tenant safe (RLS, no cross-org leak via widget token)

**Non-goals (v1)**
- Voice/video calls
- File/image upload (placeholder for v1.1)
- Co-browsing / screen share
- Multi-agent human handoff between visitors (handoff to human is the same as SMS/email ŌĆö `escalate-conversation`)
- Visitor-side typing indicators (serverŌåÆclient ping is fine; clientŌåÆserver can be v1.1)

## 3. Locked decisions

**Indexes**
- `idx_webchat_threads_widget_visitor` on `(widget_id, last_seen_at DESC)` ŌĆö for "active visitors" view
- `idx_webchat_threads_conversation` UNIQUE on `(conversation_id)` ŌĆö already implied by FK+UNIQUE; explicit for clarity
- `idx_webchat_widgets_org` on `(organization_id)`

**D. RLS** (appended to migration 005)

Same pattern as existing tables (`organization_id IN (SELECT user_org_ids())`) for SELECT/INSERT/UPDATE/DELETE on `webchat_widgets` and `webchat_threads`.

**Critical**: The browser widget never writes to these tables directly ŌĆö it goes through the `webchat-*` Deno functions which use the **service role key**, bypassing RLS. RLS only protects from the in-app (anon/authenticated) paths. This is the same pattern used for SMS/email webhooks.

## 5. New services in support-core

### 5.2 `InboundMessageService` (extend)

## 6. New function entrypoints (InsForge Deno Functions)

All under `insforge/functions/webchat-*`. All use the existing `_shared/create-db-client.ts`. None use `_shared/verify-jwt.ts` ŌĆö webchat visitor auth is separate.

### 6.1 `webchat-thread-init` (POST, public, rate-limited)

**Auth**: The widget passes `x-widget-token` (the public `webchat_widgets.widget_token`). We look up the widget and use its `hmac_secret` to verify the call's `Origin` header against `allowed_domains`. If `allowed_domains` is empty, allow.

**Request**
```json
{
  "page_url": "https://example.com/pricing",
  "referrer": "https://google.com",
  "user_agent": "Mozilla/5.0 ...",
  "pre_chat": { "name": "Sam", "email": "sam@acme.com" }
}
```

**Response**
```json
{
  "status": "ok",
  "data": {
    "visitorToken": "<jwt>",
    "threadId": "uuid",
    "conversationId": "uuid",
    "contactId": "uuid",
    "history": [
      { "id": "...", "body": "...", "sender_type": "ai", "created_at": "..." }
    ]
  }
}
```

**Server logic**:
1. Resolve widget from `x-widget-token`; reject if inactive.
2. Origin allowlist check.
3. Optional pre-chat: if `pre_chat.email` provided, find or create contact by email; else create anonymous contact.
4. Create conversation (`channel='webchat'`, `status='open'`).
5. Create `webchat_thread`.
6. Sign a visitor JWT (HS256) with claims `{ sub: contactId, org: orgId, widget: widgetId, thread: threadId, jti: randomUuid, iat, exp: iat+24h }`. Secret = `widget.hmac_secret`.
7. Return token + thread + last N messages of conversation (if any) for thread resumption.

### 6.2 `webchat-inbound` (POST, visitor JWT)

**Auth**: `Authorization: Bearer <visitorToken>`. Verify HS256 with the widget's `hmac_secret` (loaded from DB by `widgetId` claim). Reject if `jti` doesn't match the thread's current `visitor_token_jti` (rotation enforcement).

**Request**
```json
{ "text": "Hi, do you ship to Canada?" }
```

**Response**: identical to `sms-inbound` shape: `{ status, data: { message, conversationId, ŌĆ” } }`.

**Side effects**: identical to `sms-inbound` minus the org-lookup step: insert inbound message, update `last_message_at`, enqueue `process_ai_message` job, publish `new_message` on `org:{orgId}`, write audit log entry.

**Anti-flood** (lightweight, in-memory per worker): max 10 messages / minute per `threadId`. Over the limit ŌåÆ 429.

### 6.3 `webchat-session-info` (GET, visitor JWT)

Returns `{ thread, contact, history }` for thread resumption. Used by the iframe page to render prior messages on reload.

### 6.4 `webchat-identify` (POST, visitor JWT)

Body: `{ email, name? }`. Calls `WebchatThreadService.identifyThread`. Returns the new visitor JWT; the iframe replaces its stored token and continues.

### 6.5 `webchat-typing` (POST, visitor JWT) ŌĆö optional v1.1
- Body: `{ typing: boolean }`
- Realtime publish `widget:{widgetId}:{jti}`, event `typing`

(Mark as optional ŌĆö most v1 widgets can ship without this.)

## 7. New visitor-auth helper

Add `insforge/functions/_shared/verify-visitor-jwt.ts`:

```ts
export interface VisitorClaims {
  contactId: string;
  organizationId: string;
  widgetId: string;
  threadId: string;
  jti: string;
}
export async function verifyVisitorJwt(
  req: Request, db: DatabaseClient,
): Promise<{ claims: VisitorClaims; widget: WebchatWidget; thread: WebchatThread } | null>
```

Implements HS256 verification (using Node/Deno's `crypto.subtle`), looks up the widget, then the thread, then asserts `thread.visitor_token_jti === claims.jti`. Returned tuple is what function entrypoints consume.

> Why hand-rolled HS256 rather than the existing `verify-jwt.ts`? Because visitor JWTs are signed with **per-widget secrets** (not a single InsForge auth secret), and the `sub` claim is a `contactId`, not a user. Mixing the two paths would be confusing.

## 8. Embeddable widget bundle

### 8.1 Distribution

Serve the widget JS as a Next.js static asset at `/widget.js`. Build with Vite from a new `widget-src/` folder, output to `public/widget.js`. No runtime dependencies.

### 8.2 Bundle contract

```html
<script src="https://app.inboxpilot.com/widget.js"
        data-widget-id="wt_abc123"
        data-position="bottom-right"
        data-color="#2563eb"></script>
```

The script:
1. Reads `data-widget-id`. If missing, no-op + console.warn.
2. Resumes an existing session from `localStorage['inboxpilot:visitorToken:<widgetId>']` if present.
3. Otherwise calls `webchat-thread-init` to mint a fresh token.
4. Mounts an iframe pointing to `https://app.inboxpilot.com/wchat/<widgetId>?t=<visitorToken>` (avoid script + style collisions on the host page). The iframe loads a tiny page in `app/wchat/[widgetId]/page.tsx`.
5. iframe content uses the **InsForge Realtime SDK** (anonymous) to subscribe to `widget:{widgetId}:{jti}` and render messages.
6. Send: `POST` to `/functions/v1/webchat-inbound` with the visitor JWT.

### 8.3 Why iframe, not shadow DOM

- Full CSS isolation for free
- Lets the agent-side inbox re-use the existing `MessageBubble` component as a server-rendered iframe body (faster TTI on slow connections)
- Avoids `Content-Security-Policy` conflicts on host pages

The iframe page is rendered by the existing Next.js app at `app/wchat/[widgetId]/page.tsx` ŌĆö a single, fully-isolated page (no app shell, no auth gate). It only fetches messages via realtime + the `webchat-inbound` function.

### 8.4 Public surface to verify on the host page
- Bundle never executes if `data-widget-id` is invalid (no init call to backend)
- CORS: all `fetch` calls go to the InsForge base URL; widget must include `credentials: 'omit'` (no cookies)
- No third-party requests at all

## 13. Acceptance criteria

- An org owner can create a widget at `/settings?tab=webchat` and copy a `<script>` snippet.
- Pasting the snippet on a page at an allowed origin shows a chat bubble in the bottom-right with the configured color and greeting.
- A visitor can send a message. The message appears in the agent inbox within ~1s (realtime push).
- The AI drafts a reply (when org AI mode is `draft_only`) and the agent can approve it; the reply appears in the visitor's widget within ~1s.
- A `webchat_threads` row exists with `visitor_token_jti`, `page_url` updated, and `identified_at` set only after the visitor shares their email.
- Hitting `webchat-inbound` with no JWT returns 401; with an old (rotated) token returns 401; with a token from a different widget returns 401.
- Hitting `webchat-thread-init` with an `Origin` not in `allowed_domains` returns 403.
- RLS: an authenticated user not in the widget's org cannot SELECT the widget or its threads via the anon key.
- `npm test` passes; `npm run lint` passes; the `widget-src` build emits a `public/widget.js` under 40 KB gzipped.
