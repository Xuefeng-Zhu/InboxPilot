P1 · Inbox UI states · hand-off

Inbox UI states — design hand-off

Five components, twenty frames, one design language. The inbox is where a support agent lives for eight hours a day — every state has to be intentional. Loading is honest, error is actionable, escalation never gets lost, the AI never pretends to be faster than it is. This document is the contract; the inbox-states.html frames are the proof.

Contents 1. The Escalated lane 2. ConversationList 3. MessageThread 4. AiDraftPanel 5. ReplyComposer 6. StatusBadge 7. Keyboard shortcuts 8. Eng hand-off

01The Escalated lane

The escalated conversation is the most expensive thing on the screen. It costs SLA time, it costs a human's attention, and it's the single most likely place a customer is going to leave. The "Escalated" lane lives as a filter chip at the top of ConversationList, with three distinct visual states below.

Treatment decision

  • Filter chip at the top of the rail, not a separate column. Agents need the full conversation list to triage; a dedicated column would force them to context-switch.
  • Default sort: last_message_at DESC. When the Escalated chip is active, sort by escalated_at DESC instead — the most recent escalation floats to the top, even if a customer replied minutes later.
  • Color: --terra (#A8331F), never terra on a colored fill, only on a 2 px left border + terra text in the badge. Terra is for attention, not for wallpaper.
  • Badge label stays as "Escalated". We considered "Urgent" but the word is louder than the action — agents should treat escalation as do it now, not panic.
  • No badge with a count on the chip — count is on the chip itself, in mono, so the eye learns the rhythm.

ENG NOTE — Escalated lane

ConversationList needs a filter state ('all' | 'open' | 'pending' | 'escalated' | 'resolved') and a sort state (default last_message_at DESC; when filter === 'escalated', switch to escalated_at DESC NULLS LAST). The chip's is-active.escalated class paints the chip in terra; the rail item's is-escalated class paints the 2 px left border in terra. SLA countdown is escalated_at + sla_minutes minus now, rendered in mono, in --ink-3 until < 15 min, then in --terra.

04AiDraftPanel

The AI panel has the most states. The most important rule: no spinner on the thinking state. AI thinking is open-ended — a spinner with no progress is a lie. Instead: skeleton lines, an elapsed-time label, and a quiet pulse on a single dot.

AI-draft pending — design decision

  • Skeleton, not spinner. Three lines, ~92% / 78% / 64% width, animating left-to-right (shimmer). After 5 seconds, the panel collapses to a single subdued line: "Drafting… 0:05 elapsed" — the eye learns that the AI is working, then forgets about it.
  • Elapsed-time label updates every second, in mono, in --ink-3. Format M:SS for under a minute, H:MM:SS after. Stays visible the whole time — never replaced with a spinner, never hidden.
  • No "Draft ready" toast. The draft appears in place, with a 1 px teal border to signal it's new. The eye is already on this part of the screen.
  • Confidence is shown as a small mono label, never a colored bar. 0-49% is a terra outline ("low confidence — review carefully"), 50-74% is ochre, 75-100% is forest. Never trust the AI silently.
  • Keyboard: Tab from composer moves to Approve, then to Regenerate. Cmd+Enter approves when the AI panel is focused.

● AI Thinking ● AI Draft ✓ Auto-replied ● Needs Human ● AI Failed
The leading dot is a pulse animation only for thinking. For all other states, the dot is static — the action is done.

Paired (status + AI)

Open● AI Draft Pending ● AI Thinking Escalated ● Needs Human Resolved ✓ Auto-replied
Status badge first, AI badge second. They never collapse into a single glyph — two badges = two ideas, and the agent needs both at a glance.

Hover & focus

Open Focused badge — teal outline, 1px offset, never a fill change. Same focus ring style across the whole product.

07Keyboard shortcuts

Shortcuts are scoped to the inbox focus zone. When the message thread is focused, J/K navigate the list and R focuses the composer. When the composer is focused, ⌘⏎ sends and Esc leaves. Outside the inbox, no shortcut fires — we never steal global keystrokes.

Shortcut Action Scope When
J Next conversation inbox · list focused Moves focus + selection down. Wraps to first row at end.
K Previous conversation inbox · list focused Moves focus + selection up. Wraps to last row at top.
↑ / ↓ Next / prev conversation inbox · list focused Same as J/K; included for muscle memory.
R Focus reply composer inbox · thread focused Caret placed at end of existing draft or empty position.
Shift+R Ask AI to draft a reply inbox · thread focused Triggers AI draft flow even when a draft already exists. Existing draft is preserved as previous version.
⌘ ⏎ / Ctrl ⏎ Send reply composer focused Disabled when composer is empty or already sending. Triggers the same handler as the Send button.
⇧ ⏎ Newline in composer composer focused Default textarea behavior. Never sends.
E Escalate inbox · thread focused Prompts for a reason (one-line textarea), then sets status = escalated and notifies the team channel. Cannot escalate an already-resolved conversation without a confirm step.
S Resolve inbox · thread focused Sets status = resolved, archives from default rail. Sends a brief "Anything else?" auto-reply only if AI is in auto_reply mode and the org has it enabled.
Esc Leave current focus zone any Composer → thread → list → page header. Never closes the page.
/ Focus search inbox Slashes into the search input; Esc returns focus to the list.
? Show shortcuts help inbox Opens a modal with the table below. Closes on Esc or any other key.
1–5 Switch filter chip inbox · list focused 1=All, 2=Open, 3=Pending, 4=Escalated, 5=Resolved. Mirrors the chip order on screen.

08Eng hand-off

Frame map (for the [P1] Inbox UI end-to-end eng card)

All 20 mock states are in this single HTML file. Open inbox-states.html in a browser — no build step. Frame IDs are stable and referencable: ESC-1a/b/c, CL-1/2/3/4, MT-1/2/3/4, AI-1/2/3/4/5/6, RC-1/2/3/4/5/6, SB-1, KBD-1.

Component-by-component hand-off

ConversationList — frames CL-1…CL-4, ESC-1a/b/c

  • Loading uses .cl-skel with 6 skeleton items. Skeletons match the real .rail-item vertical rhythm (≈64 px tall) so there's no jump when content arrives.
  • Error keeps a cached list visible below the error box (see CL-2). Retry button is primary. "View cached" is secondary, never the only option.
  • Empty state on first load (CL-3) is distinct from "filter returned 0 results" — use a different copy and a "Show all" link.
  • Hover, focus, and selected are independent: hover:bg-surface-3, focus = 2 px teal ring inset, selected = teal-soft bg + 2 px teal left border. Never combine hover + selected; selected wins.
  • Escalated item: is-escalated class paints the 2 px left border in --terra. The badge color stays consistent across all three states.

MessageThread — frames MT-1…MT-4

  • Loading bubbles are alternating (contact / user / contact) so the user sees the conversation shape, not just a wall of gray. Width 76% / 40% / 76% — matches typical content ratios.
  • Empty state never says "Nothing here". It says "No messages yet — the customer's first message will appear here when it arrives." Time-bound, not negative.
  • Error state always mentions the draft is preserved. The single most important thing the agent needs to know: "did I lose my work?" Answer: no.
  • AI messages in the thread get a .bubble.ai style (teal-soft bg, teal border, never teal text). They sit on the contact side (left) — the AI is talking on behalf of the system, not the agent.

AiDraftPanel — frames AI-1…AI-6

  • AI-1 (thinking) uses a 3-line skeleton + elapsed-time label, no spinner. The elapsed label updates every 1s, mono, --ink-3. After 30s, switch to --ochre. After 60s, switch to --terra and add a "stuck? regenerate" inline action.
  • AI-2 (drafted): confidence is always shown, never hidden. 75-100% = forest outline, 50-74% = ochre outline, 0-49% = terra outline with a "review carefully" hint. The button is "Approve & send" (verb + delivery confirmation), not just "Approve".
  • AI-3 (escalated): this is a human handoff, not an error. The panel uses terra for attention but the icon is a ! in a terra circle, not a warning triangle — the agent needs to feel "take over", not "something broke".
  • AI-4 (failed): always include the request id in mono for support escalation. Provide three actions: Retry draft, Write manually, Report bug. Never trap the agent.
  • AI-5 (loading) and AI-1 (thinking) look similar but serve different purposes. Thinking = the AI hasn't produced anything yet. Loading = there is a draft on the server and we're fetching the body. The label should distinguish: "AI thinking" vs "Loading AI draft".
  • AI-6 (idle) is the only state where the panel can render nothing — but if we render the "press Shift+R to ask the AI" hint, we teach the agent a shortcut.

ReplyComposer — frames RC-1…RC-6

  • Composer is always 320 px tall max, scrolls internally if the user pastes a long message. Never grows taller than the thread itself.
  • The ⌘ ⏎ hint on the Send button is always visible (even on hover, even when disabled) — it's a learnable affordance, not a tooltip.
  • RC-4 (success): after sending, the composer is replaced by a single confirmation line for 1.5s, then a "Resolve" CTA appears. The agent's next action is "do I close this or wait for a reply?" — we surface both options.
  • RC-5 (error): the draft is always preserved. The error message includes a retry counter ("will retry every 5s for the next 2 min") so the agent knows the system is still trying.
  • RC-6 (disabled): when the AI is in auto_reply mode, the composer is disabled and italicized. Esc takes over manually. This is a power-user path, not a daily one.

StatusBadge — frames SB-1

  • Two separate components, <StatusBadge status="…" /> and <AiStateIndicator aiState="…" />. Never merge.
  • Badges use the same Linen token palette as the rest of the product. The dot is 5×5 px, 999 px radius. Font is IBM Plex Sans 11 px medium.
  • Hover state: 2 px teal outline at 1 px offset. Same focus style as everywhere else in the product.

Keyboard shortcuts — frame KBD-1

  • Implement in a single useInboxShortcuts hook attached to the inbox page, not at the document level. Scoping prevents stealing keys from modals, the composer, and input fields.
  • The ? overlay is a focus-trapped modal that closes on Esc or any other key. It uses the dark --ink panel — the same treatment as the analytics page's "no data" states.
  • J/K and ↑/↓ do the same thing. Implementing both is cheap and respects muscle memory from Gmail, Linear, and Front.

What this hand-off is not

This is not a Figma file. It's a self-contained HTML document with 20 frames and a spec attached. Eng should be able to take any frame ID (e.g. AI-1) and find the exact pixel treatment for that state in this file. If a frame needs to change, the change lives in inbox-states.html first, then in code, not the other way around — the design file is the source of truth for visual decisions.
